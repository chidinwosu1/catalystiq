# Deploying Catalyst IQ to Render

This describes a fresh deployment of the **current** app as two new Render
services plus a Postgres database, driven by [`render.yaml`](./render.yaml).
It does **not** touch your existing Alpaca service.

Order submission stays **disabled** and Webull stays **read-only** ŌĆö the
relevant env vars are intentionally left unset (secure defaults live in code).

---

## What gets created

| Resource | Render type | Purpose |
|---|---|---|
| `catalystiq-db` | PostgreSQL | Durable storage (Bronze/Silver/Gold, sessions, ingestion runs) |
| `catalystiq-api` | Web Service (Python) | FastAPI backend (`uvicorn catalystiq.main:app`) |
| `catalystiq-web` | Static Site | Built React frontend (`frontend/dist`) |

Build/start commands, root directories, and the publish path are all set in
`render.yaml`, so you don't enter them by hand.

## Is Postgres required?

The app boots on SQLite, but Render's web filesystem is **ephemeral** ŌĆö a
SQLite file is wiped on every deploy/restart and isn't shared across
instances. So for anything beyond a throwaway look, **Postgres is required
now**, and the blueprint provisions it and wires `DATABASE_URL` automatically.
Migrations run on each boot via the start command.

## Step-by-step

### 1. Create the Blueprint
1. Render Dashboard ŌåÆ **New +** ŌåÆ **Blueprint**.
2. Connect the `chidinwosu1/catalystiq` repo and pick this branch.
3. Render reads `render.yaml` and shows the three resources. Click **Apply**.

### 2. Enter the values Render asks for
During apply, Render prompts for the `sync: false` variables:
- **`APP_PASSWORD`** (backend) ŌĆö your private Catalyst IQ login password.
  Must be **Ōēź 12 characters** and different from the auto-generated secrets.
  *The backend will not start without it.*
- **`CORS_ALLOW_ORIGINS`** (backend) and **`VITE_API_BASE_URL`** (frontend)
  depend on the URLs Render is about to assign ŌĆö leave them blank for now and
  set them in step 4.

`SESSION_SECRET` and `ACTION_API_KEY` are generated by Render automatically ŌĆö
you don't create or paste them, and they never appear in the repo.

### 3. Let the services build
Render creates the DB, then the two services. Note the assigned URLs, e.g.:
- backend: `https://catalystiq-api.onrender.com`
- frontend: `https://catalystiq-web.onrender.com`

The backend may fail its first boot until step 4 (CORS) ŌĆö that's expected.

### 4. Wire the two URLs together, then redeploy
- **`catalystiq-api` ŌåÆ Environment:** set
  `CORS_ALLOW_ORIGINS = https://catalystiq-web.onrender.com` (your exact
  frontend URL, no trailing slash). Save (it redeploys).
- **`catalystiq-web` ŌåÆ Environment:** set
  `VITE_API_BASE_URL = https://catalystiq-api.onrender.com` (your exact
  backend URL). Save, then **Manual Deploy ŌåÆ Clear build cache & deploy** so
  the value is baked into the static build.

### 5. Sign in
Open the frontend URL ŌåÆ **Sign in** ŌåÆ enter your `APP_PASSWORD`. The NYSE
calendar and (with internet egress on Render) live Yahoo quotes should work
immediately.

## Environment variables

### Required (set/auto for a working deploy)
| Var | Where | Notes |
|---|---|---|
| `ENVIRONMENT` | api | `production` (in `render.yaml`) ŌĆö turns on the auth fail-fast |
| `APP_PASSWORD` | api | **You set it.** Login password, Ōēź12 chars, distinct |
| `SESSION_SECRET` | api | **Render generates.** Signs cookies only |
| `ACTION_API_KEY` | api | **Render generates.** Programmatic bearer token |
| `SESSION_COOKIE_SECURE` | api | `true` (in `render.yaml`) |
| `SESSION_COOKIE_SAMESITE` | api | `none` (in `render.yaml`) ŌĆö cross-site subdomains |
| `CORS_ALLOW_ORIGINS` | api | **You set it** to the frontend URL (step 4) |
| `DATABASE_URL` | api | **Wired from `catalystiq-db`** by `render.yaml` |
| `VITE_API_BASE_URL` | web | **You set it** to the backend URL (step 4) |

### Optional (only if you want that data source; add in the dashboard)
All off by default. Enabling a source with a missing key makes the backend
fail startup, so set the flag **and** its key together.
| Source | Vars |
|---|---|
| SEC EDGAR | `ENABLE_SEC_EDGAR=true`, `SEC_USER_AGENT="Catalyst IQ you@example.com"` |
| FRED (ephemeral macro panel) | `ENABLE_FRED=true`, `FRED_API_KEY=ŌĆ”` ŌĆö isolated, no-store, allowlisted public-domain series only; see [`FRED_COMPLIANCE.md`](./FRED_COMPLIANCE.md) |
| BLS | `ENABLE_BLS=true`, `BLS_API_KEY=ŌĆ”` |
| BEA | `ENABLE_BEA=true`, `BEA_API_KEY=ŌĆ”` |
| Twelve Data (restricted personal-use) | `ENABLE_TWELVE_DATA=true`, `TWELVE_DATA_API_KEY=ŌĆ”` ŌĆö central credit limits, auto-shutoff, no raw-value persistence; single-user only; see [`TWELVE_DATA_COMPLIANCE.md`](./TWELVE_DATA_COMPLIANCE.md) |
| FINRA, Nasdaq Trader | *(enabled by default; keyless)* |
| Webull (READ-ONLY) | `ENABLE_WEBULL=true`, `WEBULL_APP_KEY/SECRET/ACCOUNT_ID` |

Tunables with sensible defaults: `SESSION_TTL_SECONDS` (43200),
`SESSION_COOKIE_NAME` (`ciq_session`), `PROVIDER_COMPARISON_TOLERANCE_PCT`,
`MARKET_DATA_FALLBACK_ENABLED`.

### Do NOT set (keeps the trading safeguards intact)
`ENABLE_PAPER_ORDER_SUBMISSION`, `ENABLE_LIVE_ORDER_SUBMISSION`,
`ORDER_CONFIRMATION_SECRET`, `TRADING_MODE=live`. Leaving these unset keeps
paper-order submission disabled and live trading unavailable.

## Running the ML training dry-run (one-off)

The ML subsystem ships **disabled**. To check whether the wired point-in-time
features + available history are sufficient to train (a *dry run* ŌĆö it fits
only **candidate** models, never approves or serves anything), run the offline
runner. It reads/writes the app's own validated Silver.

**Provider & credentials.** `--ingest` uses **Yahoo Finance (`yfinance`)** ŌĆö
the only provider `get_market_data_provider()` returns. It needs **no API key
and no credentials**, only outbound network access to Yahoo. The single env var
that matters is **`DATABASE_URL`** (already set on the `catalystiq-api`
service). `numpy`/`scikit-learn` are in `requirements.txt`, so fitting works in
the deployed image.

**Keep production disabled.** Do **not** set `ENABLE_ML*` in the service
environment. The runner turns training on for *its own process only* via a
CLI flag (`--enable` / `--enable-training`); this never persists and never
enables inference, serving, or approval.

Two runners are available:

```bash
# Lightweight sufficiency check (prints a DryRunReport JSON; exits non-zero
# if the data is not yet sufficient):
python -m catalystiq.ml.dry_run_cli \
  --symbols AAPL,MSFT,NVDA,JPM,XOM --benchmark SPY \
  --start 2020-01-01 --end 2021-06-30 --horizon 5 --enable --ingest

# Fuller runner with MLflow experiment tracking + candidate registration:
python -m catalystiq.ml.train_cli \
  --symbols AAPL,MSFT,NVDA,JPM,XOM --benchmark SPY \
  --start 2020-01-01 --end 2021-06-30 --horizons 1,5,20 \
  --enable-training --ingest --register-candidates
```

**Where to run it (pick by plan):**

- **Any box with the repo + Yahoo access, pointing at the Render Postgres
  (works on the free plan).** Render's free tier has **no Shell and no Jobs**,
  so run it locally/in CI and point at the DB with `--database-url "<Render
  EXTERNAL Postgres URL>"` (or omit it and use a throwaway local sqlite via
  `--database-url sqlite:///./ml_dryrun.db` if you don't want to write Silver
  into prod).
- **Render Shell** on `catalystiq-api` (paid instance types): open the Shell
  tab and run the command as-is ŌĆö it inherits `DATABASE_URL`, the deps, and
  Yahoo egress.
- **Render one-off Job** (paid): in the dashboard, create a **Job** on the
  `catalystiq-api` service and enter the command. A Job reuses the service's
  image and environment and runs once ŌĆö no new network surface. (This is a
  dashboard/API run of the existing service, not a `render.yaml` resource; a
  *recurring* cron job or an admin HTTP endpoint would both be the wrong tool
  here ŌĆö avoid them.)

**Sizing.** Feature extraction is ~1s/example (heavy analysis snapshots). Five
symbols over ~18 months weekly is ~400 examples Ōēł 7ŌĆō8 min CPU plus model
fitting ŌĆö tight on a free 512 MB instance. Start with **3ŌĆō5 symbols and a
shorter range** (or a larger `--step-days`) and scale up.

## Caveats

- **Cross-site cookies.** Two separate `onrender.com` subdomains are
  cross-site, so the app is configured with `SameSite=None; Secure`. Modern
  browsers increasingly restrict third-party cookies; if a login doesn't
  persist, the robust fixes are (a) put both services behind one custom
  domain (`app.example.com` + `api.example.com`), or (b) serve the frontend
  from the backend so they share an origin (then `SameSite=lax`). Ask and this
  can be wired up.
- **Free tier.** Free web services cold-start after inactivity, and free
  Postgres expires after ~90 days ŌĆö fine for evaluation, upgrade for anything
  real.
